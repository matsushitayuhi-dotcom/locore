# Locore — Product Requirements Document (PRD)

**バージョン**: 0.4
**最終更新**: 2026-05-06
**ステータス**: Draft

---

## 0. ドキュメント概要

本書は、在外邦人向け旅行情報マーケットプレイス **Locore（ロコレ）** のプロダクト要求仕様書である。MVP（Phase 1）の機能要件を中心に、ビジョン、ターゲット、差別化要因、機能仕様、技術要件、ロードマップ、リスクを定義する。

---

## 1. プロダクトビジョン

### 1.1 サービス名

**Locore（ロコレ）**

語源：`local` + `lore`（伝承・語り）の造語。「現地民が語る、その土地の本当の物語」を意味する。

### 1.2 ビジョン・ステートメント

> **観光客向けではない、現地に住む日本人だからこそ知る「本物の旅情報」を、書き手の収益源にしながら旅行者に届ける。**

映え一辺倒の TikTok / Instagram に飽きた中〜上級旅行者と、現地での知見を持ちながらマネタイズ手段がない在外邦人を、信頼性ある記事マーケットプレイスで接続する。

### 1.3 ブランドトーン

- モダン・洗練
- 控えめなプレミアム感（高級ブランドではなく、知的で上質なメディア）
- 秘密感・没入感・ワクワク感
- 「映え禁止」を明文化することがブランドになる

## 2. 解決する課題

### 2.1 旅行者側（買い手）の課題

| 課題 | 既存ソリューションの限界 |
|------|----------------------|
| TikTok / Instagram は浅く、映え偏重で実体と乖離 | アルゴリズムが映え画像を優先、深い情報が埋もれる |
| ガイドブックは情報が古く、店舗忖度で本音が出ない | 出版サイクルが2-3年、広告依存 |
| 個人ブログは検索性が悪く、信頼性の判断が難しい | 書き手の現地居住歴が不明、SEO 操作も多い |
| 食べログ・トリップアドバイザーは観光客評価で「現地民が行く店」が分からない | 評価軸が単一、現地民の声が薄まる |

### 2.2 書き手側（在外邦人）の課題

| 課題 | 既存ソリューションの限界 |
|------|----------------------|
| 現地で蓄えた情報を金銭化する手段が乏しい | note は旅特化ではなく埋もれる、ブログはアフィリエイト依存 |
| TikTok / YouTube は登録者数が必要、即収益化できない | 規模ゲーム、長期戦 |
| 専門領域（パリの隠れビストロ等）の情報需要にマッチする場がない | 既存メディアは網羅性重視で深さが足りない |

## 3. ターゲットユーザー

### 3.1 買い手（旅行者）

**プライマリーペルソナ：旅行中〜上級者「アヤカ（32歳）」**

- 年収500-800万円、首都圏在住、語学に抵抗なし
- 海外渡航経験10カ国以上、フランスは2回目以上
- 「ガイドブックの王道」に飽きており、ローカルな体験を求める
- Instagram のリールは見るが、映え一辺倒には冷めている
- 旅行前に2-3週間かけてリサーチする

**サブペルソナ**：

- 新婚旅行・記念旅行で「特別感」を求める層
- リピーター（同じ国に複数回行く層）
- 駐在員の家族訪問層

### 3.2 書き手（在外邦人）

**プライマリーペルソナ：パリ駐在5年「ジュンコ（38歳）」**

- 駐在員の妻、現地在住5年、パリの食・カフェ事情に詳しい
- ブログを趣味で運営しているが収益化していない
- 副業で月3-10万円稼げるなら本格的に書きたい
- TikTok / Instagram のフォロワーは1,000-5,000人

**サブペルソナ**：

- 現地就職組（バイリンガル人材）
- 留学生（在住1-2年）
- 国際結婚で永住している層
- TikToker / YouTuber（在外邦人系クリエイター、フォロワー1万-10万）

## 4. 差別化要因（コアバリュー）

| # | 差別化要素 | 競合比較 |
|---|----------|---------|
| 1 | **旅行特化** | note は汎用、ここは旅のみ |
| 2 | **在外邦人による日本語コンテンツ** | TripAdvisor は多言語混在、深さに欠ける |
| 3 | **居住歴認証によるランク制度** | どのプラットフォームにも無い |
| 4 | **ローカル度 × 訪問満足度の2軸評価** | 既存は単一評価軸 |
| 5 | **マップ × フィードのハイブリッドUI** | note・blog はフィードのみ |
| 6 | **Google Maps連携で「持って歩く」体験** | 既存メディアは「読む」止まり |
| 7 | **観光客密度ヒートマップ** | 唯一の機能 |
| 8 | **TikToker / YouTuber のSNS連携と認証バッジ** | クロスプラットフォーム導線 |

## 5. プロダクトスコープ

### 5.1 In Scope（提供する）

- 記事（テキスト + 画像 + 動画）の作成・販売・購入
- 居住歴認証
- 評価システム（ローカル度・訪問満足度）
- フィード型・マップ型ディスカバリー
- トリッププランナー（記事のスポットを束ねて旅程化）
- Google Maps エクスポート
- SNS連携
- 編集チームによる特集
- 一般ユーザーによるライト旅行記投稿（無料）
- クライシス情報バナー

### 5.2 Out of Scope（提供しない）

- 予約代行・手配代行（旅行業法回避のため）
- 自社内での宿泊・交通機関予約
- 現地ガイド派遣
- 旅行ツアーの企画販売
- リアルタイム1on1コンサル（Phase 1 では非搭載）
- オーディオウォーキングツアー（ユーザー判断で除外）
- 鮮度管理（記事の自動アーカイブ）→ 購入者レビューで補完

## 6. 機能要件（Phase 1: MVP）

### 6.1 ユーザー管理

#### 6.1.1 アカウント種別

- **旅行者アカウント（Reader）**: 記事閲覧・購入・評価
- **書き手アカウント（Writer）**: 記事作成・販売 + Reader機能
- **編集者アカウント（Editor、内部用）**: 特集編集・モデレーション

#### 6.1.2 サインアップ

- メールアドレス + パスワード
- Google / Apple ソーシャルログイン
- 旅行者は最小限の情報で開始可能
- 書き手は追加で「居住歴認証フロー」へ誘導

### 6.2 居住歴認証（書き手）

書き手のランクを3階層で管理する。

| ランク | 提出書類 | 表示バッジ | 価格上限 | 検索順位重み |
|------|---------|----------|---------|-----------|
| **Tier S（認証クリエイター）** | 滞在ビザ + 公共料金請求書 + 現地銀行口座 | 「認証 / 居住○年」 | 上限なし | 最高 |
| **Tier A** | 滞在ビザのみ | 「居住確認済」 | ¥3,000/記事 | 中 |
| **Tier B** | 自己申告 + プロフィール審査 | （なし） | ¥1,000/記事 | 低 |

**追加**: SNS API連携で TikTok / Instagram / YouTube のフォロワー数1万人以上の場合、自動的に「クリエイター認証」サブバッジを付与（Tier とは独立）。

### 6.3 記事作成・販売（書き手）

#### 6.3.1 記事フォーマット

- タイトル
- カバー画像
- 本文（リッチテキスト：見出し、段落、画像、動画、引用、リスト）
- 動画埋め込み（TikTok / Instagram / YouTube URL）
- スポット情報（必須・複数可）：
  - 店舗名・施設名
  - 住所（Google Places API で自動補完）
  - 緯度経度
  - カテゴリ（飲食 / 観光 / ショッピング / 宿泊 / その他）
  - 料金目安
  - 営業時間
  - 訪問者向けメモ（予約必要、日本語通じる、現金のみ等のタグ）
- 価格設定（¥300〜上限はTierにより変動）
- タグ（テーマ：朝食、夜遊び、雨の日、デート等）
- 想定所要時間（半日 / 1日 / 数時間 / その他）

#### 6.3.2 公開前ガイドラインチェック

AI判定で以下を自動チェック（警告のみ、強制ブロックはしない）：

- 「映え過剰」スコア：閾値超過で警告
- 観光客密集スポット（エッフェル塔、ルーブル前など）の比率
- 文字数（最低500字推奨）

### 6.4 記事購入・閲覧（旅行者）

#### 6.4.1 プレビュー

未購入時は以下が公開される：
- タイトル
- カバー画像
- 著者名・ランク・居住歴
- 価格
- ローカル度・満足度・購入数・レビュー件数
- エリア（区・地区レベル、具体住所は隠す）
- 本文の冒頭 1-2 段落（フックとなる導入）
- スポット数（「5箇所紹介」とだけ）

#### 6.4.2 購入後

- 全文閲覧
- 全スポットの正確な住所・地図ピン表示
- オフライン保存（モバイルアプリ）
- トリッププランナーへの追加
- レビュー投稿権の付与

#### 6.4.3 価格設定

- ¥300 / ¥500 / ¥800 / ¥1,000 / ¥1,500 / ¥2,000 / ¥3,000 / ¥5,000 から選択
- 段階制にして比較・購入判断を簡素化

### 6.5 評価・レビューシステム

#### 6.5.1 メイン評価軸（必須）

- **ローカル度スライダー（0〜100）**: 「定番（0）⇄ 完全ローカル（100）」
- **訪問満足度（★1〜5）**: 行ってよかったか

#### 6.5.2 任意タグ評価（最大3つ選択）

- コスパ良
- 混雑少
- 日本語通じる
- 写真映え
- 一人OK
- 子連れOK
- 雨でもOK
- 夜も安全

#### 6.5.3 自由記述レビュー

- 200-1,000字程度
- 訪問日記入必須

#### 6.5.4 不正対策

- 評価可能なのは購入者のみ
- 著者は誰がどの評価をしたか見えない
- 統計的外れ値は重み下げ
- 同一購入者の複数評価は禁止

### 6.6 ディスカバリー

#### 6.6.1 フィード（デフォルト）

- カードレイアウト（縦スクロール）
- 各カードに表示：カバー画像、タイトル、著者、ランク、ローカル度バー、満足度、価格、所要時間、エリア
- フィルタ：
  - 都市・エリア
  - テーマタグ
  - ローカル度範囲
  - 満足度
  - 価格帯
  - 著者ランク
  - 任意タグ
- ソート：おすすめ（独自スコア）、新着、購入数順、満足度順

#### 6.6.2 マップビュー（切替）

- フィードと同じフィルタ条件で地図表示
- ピンの色分け：
  - 青：ローカル度高（70+）
  - オレンジ：定番
  - 紫：その中間
- ピンのアイコン：カテゴリ別（飲食 / 観光 / ショッピング / 宿泊）
- ピン拡大時：記事カードのミニ版がポップ
- **観光客密度ヒートマップオーバーレイ**（切替トグル）：
  - 地図上に観光客密集ゾーン（赤）と地元エリア（青）を重ねる
  - データソース：Google Places の人気時間帯 + 自社アルゴリズム
- 切替アイコン（フィード ⇄ マップ）は画面右上に常設

#### 6.6.3 検索

- フリーテキスト検索（タイトル・本文・スポット名・著者名）
- 都市・エリア絞り込み
- テーマタグ絞り込み

### 6.7 トリッププランナー（記事 → 旅程）

#### 6.7.1 ワークスペース

- ユーザーごとに「マイトリップ」リスト
- トリップ単位で作成（例：「Paris 2026 May」）
- トリップ属性：旅行先、出発日、帰国日、人数、メモ

#### 6.7.2 旅程組み立て

- 購入記事のスポットをドラッグ&ドロップで日付に割当
- 自由スポット（記事外）も追加可能
- 各日のビュー：
  - **タイムライン**：時間軸で配置
  - **マップ**：その日のスポットを地図表示、順路を線で表示
  - **リスト**：シンプルな一覧
- **半自動最適化（MVP）**：
  - スポット間の距離・移動時間を地図上に表示（Google Distance Matrix API）
  - 「最短順に並べ替える」ボタン1つを提供（純粋に距離ベース、時間制約なし）
  - 並べ替え後もユーザーが手動で再調整可能
  - 移動手段選択（徒歩 / 公共交通 / 車）
- 予算合計表示（為替自動計算）
- **Phase 2 で追加予定**：営業時間制約・テーマ別自動配置・観光客密集回避ルート提案

#### 6.7.3 Google Maps エクスポート

- 「Google Mapsで開く」ボタン
- 1日分の旅程を Google Maps Directions URL（複数経由地）で生成
- ユーザーが Google Maps アプリで turn-by-turn ナビ可能

#### 6.7.4 同行者シェア

- 旅程をURL発行で共有
- 閲覧専用 / 共同編集 を選択可能

### 6.8 マップUIカスタマイズ

#### 6.8.1 デザイン基調

- **モダン・洗練、ただし暗くなりすぎない**
- ベース：明るめのオフホワイト背景 + 抑えた彩度
- 道路：細く控えめに、主要道路のみ強調
- 建物：薄いグレーで立体感は最小
- 公園・水域：くすんだ緑・青
- ラベル：日本語表記優先、フォントは独自

#### 6.8.2 実装

- Google Maps Platform の **Cloud-based Map Styling** を使用
- 純正UIコンポーネントは全て非表示、自前UIで再構築
- ピンは記事サムネ or テーマ別アイコン
- 情報ウィンドウは独自カードで実装

### 6.9 SNS連携

#### 6.9.1 連携対応プラットフォーム

- TikTok
- Instagram
- YouTube
- X（旧 Twitter）
- 個人ブログ（RSS）

#### 6.9.2 機能

- プロフィールにリンク表示
- API連携で最新投稿を著者ページにフィード表示
- フォロワー数による認証バッジ自動付与
- 記事内に SNS 動画埋め込み
- アフィリエイトコード発行：SNS 経由の流入を計測

### 6.10 決済・送金

#### 6.10.1 購入者決済

- Stripe（クレジット / デビットカード）
- Apple Pay / Google Pay
- 通貨：JPY（基軸）

#### 6.10.2 書き手への送金

- 海外在住書き手への支払いは Wise または Payoneer 連携
- 国内在住書き手は通常の銀行振込
- 月次精算（毎月15日締め、月末送金）
- 最低送金額：¥3,000

#### 6.10.3 手数料

| 区分 | 手数料率 |
|------|---------|
| 認証クリエイター（Tier S または SNS 1万人以上） | **10%** |
| 通常書き手（Tier A / B） | **20%** |

→ **2倍差をつけることで認証取得への強いインセンティブ**

### 6.11 ライト旅行記投稿（無料・一般ユーザー）

旅行後の「軽い旅行記」を一般ユーザー（旅行者）が無料で投稿可能。

- フォーマット：タイトル + 写真複数枚 + 短文 + 訪問スポットタグ
- 課金なし、書き手ランク対象外
- SEO 効果（ロングテール検索獲得）
- コミュニティ感の醸成
- 公開・非公開選択可能
- スパム対策：1ユーザー月間投稿数上限

### 6.12 編集チーム特集

編集チーム（運営）が既存記事を束ねた特集を企画・公開。

- 特集テーマ例：「春のパリ：花見スポット20選」「初めてのバンコク完全プラン」
- 既存記事の束ね（記事自体は変更しない）
- 特集ページに固有のキュレーションコンテンツ + 紹介記事リスト
- 特集経由で売れた記事は **印税的に売上配分**（特集収益の10%を編集チームへ）
- フィード上部に特集枠を常設

### 6.13 クライシス情報

旅行先のリアルタイム情報を無料表示。

- データソース：
  - フランス：RATP（地下鉄）/ SNCF（鉄道）/ Le Monde 等のニュース
  - 各国の交通機関・政府発表
- 表示内容：
  - ストライキ
  - 大規模デモ・封鎖
  - 大型イベントによる混雑
  - 治安情報
- ユーザーの旅程に重ねて影響を可視化（旅行日に該当するイベントをハイライト）
- **無料機能**として広く解放、プラットフォームへの入口にする

## 8. 非機能要件

### 8.1 パフォーマンス

- フィード初期ロード: 2秒以内
- マップロード: 3秒以内
- 検索結果表示: 1.5秒以内

### 8.2 可用性

- SLA: 99.5%
- 計画停止は深夜帯（日本時間 03:00-05:00）に限定

### 8.3 セキュリティ

- HTTPS 必須
- パスワードは bcrypt ハッシュ化
- 個人情報（ビザ書類等）は暗号化保存
- PCI DSS 準拠（Stripe 経由で実現）
- 居住認証書類はアップロード後30日で自動削除

### 8.4 多言語対応

- Phase 1: 日本語 UI のみ
- Phase 3: 英語 UI 追加（書き手・国際展開向け）

### 8.5 アクセシビリティ

- WCAG 2.1 Level AA 準拠を目標

## 9. 技術スタック（推奨）

### 9.1 プラットフォーム戦略（確定）

**Phase 1（MVP）: Web (Next.js) + PWA 単一構成**

- ネイティブアプリ（iOS/Android）は **Phase 1 では作らない**
- Web をレスポンシブ + PWA 化することで、モバイルブラウザでも以下を実現：
  - ホーム画面追加（疑似アプリ化）
  - オフライン閲覧（Service Worker）
  - GPS 利用（Geolocation API）
  - プッシュ通知（Android 完全対応 / iOS 16.4+ 対応）
  - 全画面表示
  - カメラ・撮影
- App Store / Google Play 経由の課金を持たないため、**30% 手数料を回避**
- 単一コードベースで開発速度・保守性を最大化

**Phase 2 以降のネイティブアプリ判断**

以下のシグナルが出たタイミングでネイティブアプリ開発を検討：

- MAU 5万人超、PWA の限界が見え始める
- In-trip（旅行中）利用率が高く、リテンション課題になる
- バックグラウンドGPS、AR等のネイティブ機能が必要

## 10. 法務・コンプライアンス

### 10.1 旅行業法

- **情報提供のみ** に限定し、旅行業に該当しない設計を維持
- 予約代行・手配代行・ツアー販売は **In Scope外**
- アフィリエイトは「外部サイトへの誘導」として実装（媒介と見なされない表現を徹底）
- 利用規約に「Locore は旅行情報の提供のみを行い、旅行手配を行いません」と明記

### 10.2 プロバイダ責任制限法

- 通報フォーム設置（記事単位・著者単位）
- 通報対応SLA：72時間以内に一次対応
- 名誉毀損・個人情報侵害の即時削除フロー
- 開示請求対応窓口

### 10.3 景品表示法・薬機法

- 健康・医療系の記述ガイドライン制定
- 「世界一」「最高」等の最上級表現の制限
- 体験談に基づく表現であることの明示

### 10.4 個人情報保護

- プライバシーポリシー
- 居住認証書類の取り扱い明示
- 30日後自動削除

### 10.5 海外居住者への支払い

- 非居住者への支払い：源泉徴収・租税条約の確認
- 税理士監修のヘルプページを書き手向けに提供

### 10.6 利用規約

- コンテンツの著作権は書き手に帰属
- Locore はプラットフォーム利用ライセンスを取得
- ユーザー間紛争への運営介入範囲

## 13. リスクと対策

| リスク | 影響度 | 発生可能性 | 対策 |
|------|------|----------|------|
| ニワトリ卵問題（書き手 / 読み手の同時集客） | 高 | 高 | パリ単都市集中、ファウンディング特典で書き手先行確保 |
| 駐在員の任期入れ替わり | 中 | 高 | 駐在員以外（永住者・現地就職組）の比率を50%以上に |
| 記事の鮮度劣化 | 中 | 中 | 購入者レビューが事実上の鮮度シグナル、Q&A欄で補完 |
| 旅行業法の境界線 | 高 | 低 | 利用規約で明記、予約機能を絶対実装しない |
| 法的トラブル（名誉毀損等） | 中 | 中 | 通報→対応SLA設計、保険加入 |
| 決済の海外送金コスト | 中 | 高 | Wise/Payoneer 連携、月次バッチ送金で手数料圧縮 |
| 大手参入（食べログ・トリップアドバイザーが類似機能） | 高 | 中 | 在外邦人コミュニティの先行確保 + 居住認証で参入障壁 |
| 自作自演・偽レビュー | 高 | 中 | 評価は購入者限定、外れ値検知、決済アカウント連携 |
| 書き手の競合移籍 | 中 | 中 | 認証バッジの永続性、専属契約は結ばない代わりに独自機能で囲い込み |

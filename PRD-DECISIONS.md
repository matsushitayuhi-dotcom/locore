# Locore オープンクエスチョン解決提案 (#3-#6)

**バージョン**: 0.1
**最終更新**: 2026-05-06
**対象**: PRD v0.3 §15 のオープンクエスチョン #3-#6
**前提**: PRD.md v0.3、ARCHITECTURE.md v0.1

---

## 0. 全体方針

本書は Locore の MVP（Phase 1、パリ単都市集中）に向けた4つの未決事項を解く。共通する設計原則は以下：

1. **「映え禁止」をブランド資産化する** — #3 と #5 は、表現が一貫していなければただの規約違反になる。AI 判定 → 編集チームのキュレーション → 特集 までを一本の編集思想で貫く。
2. **手動 → 半自動 → 自動の3段階移行** — 在外邦人マーケットのデータは初期スパースで、機械学習や API 自動化が空回りする。最初は人手で品質を作り、データが貯まったら段階的に置き換える。
3. **質を担保する境界は「居住認証 + 購入者レビュー」に集約** — 既存の Tier S/A/B と購入者限定レビューを最大活用し、追加の参入障壁（招待制等）を作りすぎない。
4. **コスト試算と KPI の整合** — ARCHITECTURE §9 のコスト構造（M3 約¥30,000 / M12 約¥670,000）を逸脱しない範囲で施策設計する。

## #3. 「映え過剰」AI 判定の閾値設計

### 3.1 推奨

**3次元スコア × 警告ベース運用 + 編集チームの最終判断ループを併走させる。**

| 項目 | 推奨 |
|------|------|
| 判定タイミング | 「公開申請」時点（下書き保存ではなく） |
| スコア構造 | 3次元（観光地依存度 / 視覚装飾度 / 文体浅薄度）の合成スコア（0-100） |
| 閾値運用 | 70+ で **警告のみ**、85+ で **公開申請をホールド → 編集者レビュー必須** |
| 強制ブロック | **しない**（M0-6）。M7 以降、誤検知率 < 5% を確認できたら 90+ で自動ブロックを検討 |
| AI モデル | Claude Sonnet（本文）+ Claude Haiku（画像メタ）の二層構成 |
| データに基づく調整 | 月次で著者・購入者の双方からシグナルを集約、四半期ごとに閾値見直し |

### 3.2 根拠

**なぜ「ブロック」ではなく「警告 + 編集者レビュー」か：**

- Locore のブランドは「映え禁止を明文化することがブランドになる」（PRD §1.3）。これは UX で体験させるべきもので、AI が機械的にブロックすると「Tier S の永住者が書いた、たまたま定番が多い記事」が殺されてしまう。
- 在外邦人ライターのプロフィールはペルソナ「ジュンコ（38歳、駐在員の妻）」の通り、副業ライター層が中心。誤ブロック1件で離脱する確率が高い。
- 一方で「映え過剰」は Locore の唯一のブランド境界線であり、ザル化すると差別化が崩れる。**機械が一次フィルタ → 人が判断**の二段構えが最小コストで品質を担保できる。

**なぜ3次元か：**

- **観光地依存度**：エッフェル塔・ルーブル等のスポットが記事内に占める割合は構造化データ（`spots` テーブル）から正確に取れる。AI ではなく単純な集計で判定可能、誤検知ゼロ。
- **視覚装飾度**：カバー画像・本文画像の構図・彩度・人物比率を画像 AI で測る。ここは確率的判断になるが、構造化データではカバーできない領域。
- **文体浅薄度**：「最高」「絶景」「インスタ映え」「死ぬ前に行きたい」等の最上級・流行語の頻度。本文 AI で測る。10.3（景品表示法・薬機法）の最上級表現制限とも整合する。

3次元に分けることで、後述の閾値調整時に「どの次元が誤検知を生んでいるか」を切り分けやすい。

### 3.3 各次元の詳細

#### 3.3.1 観光地依存度スコア（0-100）

**入力**：記事内の `spots[]` の緯度経度、`category`、Google Places の人気時間帯データ。

**算出**：
```
observed_density = Σ (spotの観光客密度ヒートマップ値, 0-1)
spot_count = len(spots)
tourist_ratio = observed_density / spot_count
score = round(tourist_ratio * 100)
```

**観光客密度ヒートマップ** は PRD §6.6.2 で既に Phase 1 機能として定義済み。これを公開判定の入力にすることで、別データソースを増やさず再利用できる。

**閾値感覚**：
- 0-40：完全ローカル中心（マレ地区の路地裏、20区の市場等）
- 40-65：ローカル + 軽い定番ミックス（ジュンコの想定記事帯）
- 65-85：定番中心、観光客多め（要警告）
- 85-100：エッフェル塔・ルーブル・凱旋門等のみ → ホールド対象

#### 3.3.2 視覚装飾度スコア（0-100）

**入力**：カバー画像 + 本文の最初の3枚の画像。

**Claude Haiku への画像分析プロンプト（System Prompt 案）**：

```
あなたは旅行メディア「Locore」の画像審査AIである。
Locoreは「映え過剰でない、現地民視点の本物の旅情報」を価値とする。

以下の画像を5次元でスコア（各0-20、合計0-100）：

1. 構図の作為性：プロカメラマン的構図、極端なシンメトリー、SNS用ポーズ等
2. 彩度・加工度合い：HDR強調、ビビッド加工、フィルタ多用
3. 観光名所のアイコニック撮影：エッフェル塔・凱旋門等を画面中央に大きく配置
4. 人物の登場：被写体としての人物（特に著者本人がポーズ）
5. ブランド・有名アイコン依存：Louis Vuitton、Chanel等のブランド店、有名カフェの看板等

出力は JSON のみ：
{
  "composition": 0-20,
  "saturation": 0-20,
  "iconic": 0-20,
  "person": 0-20,
  "brand": 0-20,
  "total": 0-100,
  "reasoning": "1-2文の理由"
}

判定基準は厳しすぎず、迷ったら低めに付けること。
日本人の在外邦人が「ジュンコ（駐在員の妻）」のような自然なライフスタイル写真を撮るのは正常で、これは映えではない。
```

**運用上の注意**：
- カバー画像 + 最初の3枚を審査対象にする（全画像だとコストが膨らむ）
- 5次元の合計を `visual_score` とする
- 1記事あたり Claude Haiku 1回 ≒ $0.001 程度。月1,000記事公開で $1/月。

#### 3.3.3 文体浅薄度スコア（0-100）

**入力**：本文のテキスト全体（最大8,000字想定）。

**Claude Sonnet への本文分析プロンプト（System Prompt 案）**：

```
あなたは旅行メディア「Locore」の編集AIである。
Locoreは「現地に住む日本人による、深く、地に足のついた旅情報」を価値とする。
逆に、TikTok/Instagramの浅い「映え記事」を排除する。

以下の記事本文を5次元でスコア（各0-20、合計0-100。スコアが高いほど「映え記事的」）：

1. 最上級・誇張表現の濃度：「最高」「絶景」「死ぬ前に」「神」「ヤバい」「映え」等の頻度
2. 個人体験の薄さ：実際に通った頻度・店主との関係・季節変化等の「居住者ならではのディテール」が欠如
3. 価格・営業時間・予約方法等の実用情報の欠如
4. SNS文体・絵文字濫用
5. 観光客向け定型文：「初めてのパリで絶対外せない」「旅好き必見」等

出力は JSON のみ：
{
  "hyperbole": 0-20,
  "shallow_experience": 0-20,
  "missing_practical": 0-20,
  "sns_style": 0-20,
  "cliche": 0-20,
  "total": 0-100,
  "flagged_phrases": ["記事から実際に拾った懸念フレーズ最大5つ"],
  "reasoning": "2-3文で総評"
}

注意：著者の文体の好みは尊重する。砕けた口語自体は減点対象ではない。
```

#### 3.3.4 合成スコア

```
final_score = 0.4 * tourist_score + 0.35 * visual_score + 0.25 * text_score
```

- 観光地依存度を最大重み（0.4）にしているのは、`spots[]` 構造化データ由来でノイズが少ないため
- 視覚（0.35）と文体（0.25）は確率的判定なので重みを下げる
- 重みは M3 時点でデータを見て再調整する

### 3.4 警告 vs ホールドの分岐

| `final_score` | アクション | UX |
|----------|----------|----|
| 0-69 | **通過** | 何も表示しない |
| 70-84 | **警告のみ** | 公開申請画面でモーダル表示 → 著者は「修正する」/「このまま公開」を選択可能 |
| 85-100 | **ホールド** | 「編集チームが72時間以内に確認します」と通知。`articles.status = 'pending_review'` |

ホールドは「ブロック」ではなく「キュー入れ」とユーザーには見せる。心理的ハードルを下げる。

### 3.5 Tier 別運用ルール

| Tier | 70-84 | 85-100 |
|------|-------|--------|
| Tier S（認証クリエイター） | 警告のみ、公開可能 | 警告のみ、公開可能（フラグだけ立てる） |
| Tier A（ビザ確認済み） | 警告のみ、公開可能 | ホールド → 編集者レビュー |
| Tier B（自己申告） | ホールド → 編集者レビュー | ホールド + 強い修正推奨 |

これは PRD §6.2 の Tier 制度と整合する。

## #4. ファウンディングメンバーの定義

### 4.1 推奨

**先着50人 × 軽い審査制（プロフィール確認のみ）のハイブリッド。**

| 項目 | 推奨 |
|------|------|
| 定員 | **先着50人**（PRD 既定）+ 残席は M3 まで先着で埋める |
| 入口 | 招待リンク制（コミュニティ経由）+ 自薦応募フォーム |
| 審査 | **軽い審査制**：プロフィール、SNS、居住歴の自己申告で判定（書類審査なし、5分以内に運営判断） |
| 特典 | 手数料0%（6ヶ月）+ 永久バッジ + 機能優先アクセス + 編集チーム特集枠優先 + 売上トップ3名のロゴ掲載 |
| 期限 | M0-3 の3ヶ月で50人埋めることを目標、埋まらない場合は条件を緩めて期限延長 |

### 4.2 根拠

**「先着50人」を選ぶ理由：**
- PRD §11.2 で既定。希少性が明確で、コミュニケーションが楽。
- 「最初の3ヶ月の登録者」だと、無条件に何百人も入れてしまうリスク。手数料0%が累積コストになる。
- パリ単都市集中で在仏日本人界隈は狭く、50人は実態に即した規模。

**「ハイブリッド（招待 + 自薦）」を選ぶ理由：**
- 招待のみだと運営の知り合いネットワークに依存し、多様性が狭まる。
- 自薦のみだと「Locore とは何か」を理解していない人が応募し、初期 KPI が悪化する。

**「軽い審査制」を選ぶ理由：**
- ノーチェックは PRD §13 のリスク「業者の偽記事量産」を初期から呼び込む。
- 厳格な書類審査（Tier S 相当）は M0 段階では運営工数が破綻する。

### 4.3 特典詳細

| 特典 | 期間 | 詳細 |
|------|-----|------|
| **手数料0%** | 6ヶ月（M0-6） | 通常 Tier S 10%、Tier A/B 20% を 0% に |
| **「ファウンディングクリエイター」永久バッジ** | 永久 | プロフィールと記事カードに表示 |
| **居住認証の優先審査** | 永久 | Tier S 認証申請を運営が48時間以内に処理 |
| **機能優先アクセス** | 永久 | Phase 2 機能を一般公開2週間前に解放 |
| **編集チーム特集枠優先** | M0-6 | 月次特集の最初の3本にファウンディングメンバーの記事を必ず1本以上含める |
| **売上トップ3名のロゴ掲載** | 半期ごと | Locore TOP ページの「ファウンディングクリエイター」セクション |
| **オフラインミートアップ参加権** | 永久 | M3、M6、M12 のパリ現地イベントに招待 |
| **直接フィードバックチャネル** | M0-6 | プロダクトチームと月1の Discord/Slack 双方向チャネル |

### 4.4 手数料0%の累積コスト試算

| 指標 | M3 | M6 |
|-----|----|----|
| GMV | ¥100,000 | ¥1,000,000 |
| 通常時手数料収益（平均15%） | ¥15,000 | ¥150,000 |
| ファウンディング期間中の喪失収益 | ¥15,000 | ¥150,000 |

→ M0-6 の累積喪失収益は約 ¥500,000-700,000。書き手1人あたり ¥10,000-15,000 の CAC 相当。

### 4.5 質の担保：審査運用

**応募フォーム：**
1. お名前 / SNSアカウント（任意で複数）
2. 居住国・居住歴（自己申告）
3. なぜ Locore に書きたいか（200-400字）
4. 書きたいテーマを3つ
5. すでに公開している旅行関連の文章（任意）
6. ファウンディングクリエイターになって取り組みたいこと（任意）

**運営判定基準：**

| 基準 | OK | NG |
|------|----|----|
| 居住歴 | 1年以上、現地生活の具体的描写がある | 「観光で詳しい」だけ |
| テーマ | 「マレ地区のビストロ巡り」等、具体ジャンル | 「パリのおすすめ」のみ |
| 文章サンプル | 既存ブログ・SNS で5本以上の執筆実績 | テンプレ的、SNS定型句のみ |
| Locore 理解 | 「映え禁止」のブランドトーンを言語化できている | 「自分の SNS の延長で」と書く |

### 4.6 失敗時のリスク管理

| リスク | 対策 |
|------|-----|
| 50人埋まらない | M0-3 で30人未満なら M4-6 を「セカンドコホート」として再募集 |
| 50人を集めたが活動しない | 「アクティブ条件」：M3 までに最低1記事公開がない場合、バッジを「ファウンディング登録者」に降格 |
| 質の低い応募が殺到 | 募集を一旦止めて選考期間を設ける |
| 手数料0%の悪用 | AI モデレーション（#3）は通常通り適用、外れ値検知 |
| 6ヶ月後の手数料移行で離反 | 移行3ヶ月前から複数回告知 + Tier S 認証取得を支援 |
| 「ファウンディング」表記の希少性が将来失われる | 50人で締切を厳守。M6 以降の追加バッジは別名称（「アンバサダー」等） |

#### 4.7.2 手数料0%の実装

```typescript
function calculateFee(writer: WriterProfile, amount: number): number {
  if (
    writer.founding_member &&
    writer.founding_fee_waiver_until &&
    new Date() < writer.founding_fee_waiver_until
  ) {
    return 0;
  }
  if (writer.tier === 'S' || hasCreatorBadge(writer)) return amount * 0.10;
  return amount * 0.20;
}
```

## #5. 特集の編集チーム：内製 vs 外注

### 5.1 推奨

**M0-6 は内製1人（編集チーフ） + フリーランスエディター1-2名のハイブリッド。M7 以降、各都市展開に合わせて段階的にフリーランスを増やす。**

| 期間 | 構成 | 月次特集本数 |
|------|------|------------|
| M0-3 | 内製：編集チーフ1名（兼モデレーション）+ フリーランス1名（パリ在住経験者） | 月2本 |
| M4-6 | 内製：編集チーフ1名 + フリーランス2名 | 月3-4本 |
| M7-12 | 内製：編集チーフ1名 + 編集アシスタント1名 + フリーランス3-4名 | 月4-6本 |

### 5.2 根拠

- 内製のみは固定費が大きい（編集者2人で月60-100万円）。M0-6 のバーンレートに対し過大。

- ハイブリッド：内製1名がブランドガーディアン、フリーランスが実作業。

**なぜフリーランスは「パリ在住経験者」必須か：**

## #6. クライシス情報のデータソース調達

### 6.1 推奨

**M0-6 は手動キュレーション × 1日1回の編集チーム運用、M7-12 で半自動化（RSS + Twitter 監視）、Phase 2 で API 完全自動化。**

| 期間 | 体制 | データソース | 頻度 |
|------|------|------------|------|
| M0-6 | 編集チーム手動キュレーション | RATP / SNCF 公式、Le Monde、@RATPgroup（X）、在仏日本大使館 | 平日朝 1回（10:00 JST = 02:00 CET）+ 重大事象は随時 |
| M7-12 | RSS + Twitter API 半自動 | 上記 + RATP RSS、SNCF RSS、AFP RSS、X Filtered Stream | 4時間ごとバッチ + 重大度7+で即時 |
| Phase 2 | 自動化 + 編集承認 | 上記 + 政府発表 API + 自社の購読者投稿 | リアルタイム |

### 6.2 根拠

- パリのストライキ・デモは大半がフランス語ソースのみ。日本語訳・要約が必要 → 手動が現実的。
- M0-6 の MAU は1,000-5,000人。1日数件の手動更新でも運営工数は1-2時間/日で収まる。
- 自動化を急ぐと誤情報リスク。Locore は信頼性で差別化するため、誤情報1件でブランド毀損が大きい。
- 編集チーム（#5）が #3 のモデレーションと #6 のクライシス管理を兼務することで固定費を圧縮。

**なぜ「無料機能」か：**
- クライシス情報は買い手獲得のフック
- SEO 効果も大きい：「パリ ストライキ 2026年5月」
- 既存の有料記事の購入動機にもつながる

### 6.5 スクレイピングの合法性

| サイト | 可否 | 根拠 |
|------|---|------|
| RATP / SNCF / 外務省 | RSS が公式提供 → RSS 経由を推奨 | 公的機関 |
| Le Monde | 本文スクレイピングは不可、RSS 見出しのみ可 | 著作権 + 利用規約 |
| Reuters / AFP | 不可（商用ライセンス必須） | 通信社は厳格 |
| X | 公式 API 経由のみ | ToS |
| Google News | RSS 経由は可、HTML スクレイピングはグレー | RSS が公式 |

**Locore の方針：**
1. 公式 API / RSS のみ使用（スクレイピングしない）
2. 本文転載しない（見出し + 出典 + 外部リンク）
3. 編集チームが複数ソースから自分で要約を書く
4. 在仏邦人コミュニティの情報は「裏取りの参考」、表に出さない

# Locore マルチシティ展開 UX レビュー

作成日: 2026-05-18
対象: Locore (apps/web)
読者: PdM・経営判断者 / フロントエンドエンジニア
ステータス: ドラフト (UAT 前レビュー用)

---

## 1. 全体方針

Locore は「Paris に住む日本人ライター」を主軸に立ち上がったが、ボルドー在住・ニース在住・将来のリヨン/マルセイユ等を含めて **「在仏邦人ライターの集合知マーケットプレイス」** へと拡張するフェーズに入る。本ドキュメントの基本原則は次の通り。

- **Paris を特別扱いしない (表記上)**: UI/コピー/ナビゲーションのレベルで「Paris ありき」の文言を排除する。ボルドー在住者が `/explore` を開いたときに「ここは Paris の人のサイトだ」と感じさせてはならない。
- **内部実装的には Paris を先に育てる戦略は維持**: 画像素材の蓄積、AI cron (`parisEvents`)、座標バリデーション等は Paris から検証する。ただし、命名・API 設計は最初から「都市切り替え可能」を前提に汎用化しておく。
- **情報の階層**: `世界 → 国 (FR) → 都市・地域 (Paris / Bordeaux / Nice / ...)` の 3 階層ツリーで考える。当面は国 = フランス固定だが、将来の伊・西・独 展開を阻害しない命名を選ぶ。
- **UI 戦略**: グローバルな「都市セレクタ」を独立 UI として置くのは時期尚早。当面は **URL (`/region/[slug]`) と読者の居住地コンテキスト**で自然に切り替えるアプローチを採る。グローバルナビには「地域から探す」ドロップダウン程度に留め、Paris-as-default の挙動を避ける。

## 2. ペルソナ別シナリオと期待されるフロー

### シナリオ A: ボルドー在住者が新規登録 → 自分の街の記事を書く

**遷移**: `/auth/signup` → メール認証 → `/settings/profile`(居住地 = FR/ボルドー) → `/become-writer` → `/writer/articles/new` → スポット座標入力

| 段階 | 状態 |
|---|---|
| 現状の挙動 | `/settings/profile` で居住地を「ボルドー」と選択できるかは未確認。`SpotEditor` の座標バリデーションは Paris 範囲 (lat 48.x / lng 2.x) を前提にしている可能性が高く、ボルドーの座標 (lat 44.8 / lng -0.6) を入れると「範囲外」警告が出る恐れ。`/explore` のサブタイトルが「14 地域 (= Paris の 14 arrondissements?)」固定であれば、ボルドー在住者は混乱する。 |
| 気持ち悪い箇所 | 「私の街を書こうと思ったのに、フォームがパリの形をしている」 — placeholder が「6 区のカフェ」「マレ地区」等のパリ前提例だと、自分が書こうとしているサンテミリオンやサン・ピエール地区の記事が異物に感じる。 |
| あるべき挙動 | 居住地に応じて、(1) SpotEditor の座標初期値が居住都市の中心、(2) placeholder が居住都市の地名例、(3) `/explore` のサブタイトルがコンテキスト依存 (「あなたの街と、世界の街」など汎用) になる。 |

### シナリオ B: ニース在住者がボルドーへの週末旅行記事を書く (重要)

**遷移**: ニース在住として登録済みの書き手が、ボルドー旅行から戻り `/writer/articles/new` で「ボルドー、ワインと美食の週末」を執筆する。記事の `city_id` = ボルドー、書き手の `residency_city` = ニース。

| 段階 | 状態 |
|---|---|
| 現状の挙動 | 記事執筆フォームに「**自分の在住都市**」と「**この記事の対象都市**」を分けて指定する UI があるかが不明。もし「city = ユーザーの居住地」が暗黙で使われていれば、ニース在住者はボルドー記事を投稿できない、もしくは「ニースの記事」として誤分類される。 |
| 気持ち悪い箇所 | 「自分はニースに住んでるけど、この記事はボルドーの話。なんで居住地のラベルが記事に貼られるの?」 — 読者側にも「ボルドーの記事なのにライター紹介がニース」と齟齬として伝わる。 |
| あるべき挙動 | 記事フォームに **対象都市セレクタ (`article.city_id`)** を必須項目として用意し、ユーザーの `residency_city` とは独立に選べる。ライター紹介には「ニース在住、今回はボルドー旅行記」と両方を表示する。 |

### シナリオ C: 日本の旅行者がボルドー記事を読み、関連を探す

**遷移**: 日本から Google 検索で記事 `/articles/[slug]` 着地 → 読了 → 「ボルドーの他の記事」「同じカテゴリ (ワイン) の他の記事」「ボルドー街ホーム」へ遷移したい。

| 段階 | 状態 |
|---|---|
| 現状の挙動 | `/region/bordeaux` ホームページが整備されているか不明。記事下の関連記事リンクは「同じカテゴリ」で動いていても、「同じ都市」フィルタが効くかは未検証。`SiteFooter` に Paris ハードコード文言があれば、ボルドー記事を読んだ後に「Paris のサイトだったのか」と離脱する。 |
| 気持ち悪い箇所 | 「ボルドーの記事は面白かったけど、他のボルドー記事への動線がない」 — マーケットプレイス特性 (回遊性) が失われる。 |
| あるべき挙動 | 記事末尾に固定で「📍 ボルドーの他の記事」「🍷 同じカテゴリ (ワイン系) の記事」「✍️ このライターの他の記事」3 ブロックを表示。`/region/bordeaux` がボルドーの hero 画像 + 特集 + 全記事一覧の入口として機能する。 |

## 3. 「Paris 専用」現状リスト (コード監査ベース)

別エージェントの監査と `grep paris|Paris` の結果から、以下が Paris ハードコード候補。

### 高インパクト (UX に直接出る)

| ファイル | ボルドー住人視点での体験 |
|---|---|
| `apps/web/components/writer/SpotEditor.tsx` | 座標バリデーションが Paris 範囲想定なら、ボルドー座標を入れて警告が出て**記事が投稿できない**。 |
| `apps/web/components/SiteFooter.tsx` | フッターに「Paris 在住の日本人による…」等の文言があれば、全ページで違和感。 |
| `apps/web/app/articles/page.tsx` | 記事一覧のヘッダーコピーやデフォルトフィルタが Paris 寄りだと、ボルドー記事が「おまけ」に見える。 |
| `apps/web/app/admin/board/page.tsx` / `AiTestPanel.tsx` | 管理画面の都市選択が Paris 固定なら、運用側からマルチシティ運用が回せない。 |

### 中 (違和感)

| ファイル | 体験 |
|---|---|
| `apps/web/lib/ai/parisEvents.ts` | ファイル名・関数名が `paris` 固有。将来 `cityEvents(cityId)` に汎用化が必要。動作には今すぐ影響しないが、ボルドー用 cron を足すたびにファイルが増殖する。 |
| `apps/web/app/api/cron/ai-paris-events/route.ts` | 同上。URL パスに `paris` が入っており、`/api/cron/ai-city-events?city=bordeaux` 型への移行候補。 |
| `apps/web/components/PlaceMenu.tsx` / `MapInner.tsx` / `MapView.tsx` | 地図初期表示が Paris 中心固定なら、ボルドー記事を開いた時にまず Paris が映って 1 秒不快。 |
| `apps/web/lib/mock/*.ts` / `packages/db/seed/data/*.ts` | モック・シードデータが Paris のみだと、開発者・QA がボルドー UI を検証しづらい。 |

### 低 (例文・ヒント)

| ファイル | 体験 |
|---|---|
| `apps/web/lib/mock/spots.ts` / `lightDiaries.ts` / `writers.ts` 等 | placeholder やサンプルテキストに「マレ」「6 区」が散見されれば、ボルドー住人にとって「ヒントが効かない」程度の違和感。 |
| `packages/shared/src/constants.ts` | デフォルト都市定数があれば確認 (`DEFAULT_CITY = 'paris'`) し、`undefined` か `null` をデフォルトにできないか検討。 |

## 4. デザインシステムの観点

- **カバー画像 / hero image**:
  - 当面は **都市ごとに 1-3 枚の hero を手動キュレーション** (`/public/cities/bordeaux/hero-01.jpg`) し、`/region/[slug]` ホームでランダム表示する。
  - CDN/Unsplash 動的取得は将来オプション。ライセンス管理コストが上がるので Phase 3 以降。
- **配色 (Editorial Light: terracotta / cream / ink)**:
  - 都市共通で OK。Locore のブランド価値は「都市横断の編集感」なので、都市ごとに配色を変えると逆にバラバラに見える。
  - ただし、都市カードや region ヘッダーに **アクセントカラー (= 都市シンボル色) を 1 色だけ** 添えるのはアリ (Paris=ink-blue, Bordeaux=wine-red, Nice=mediterranean-blue)。
- **Emoji の使い方**:
  - 関連記事リンクや region カードで `🗼 Paris / 🍷 Bordeaux / 🌊 Nice` の 1 emoji ルールを採用する。
  - ただし本文中・記事タイトルでは抑制する (検索結果やシェア時のノイズになる)。
  - 都市セレクタの一覧では `emoji + 都市名` で視認性を確保。

## 5. URL 設計の見直し提案

現状想定される URL とマルチシティ対応の提案:

```
/                                  # 世界 (全都市横断トップ)
/region/[slug]                     # 都市ホーム (例: /region/bordeaux) ← 既存
/articles                          # 全記事一覧
/articles?region=bordeaux          # 都市フィルタ ← 追加すべき
/articles?region=bordeaux&cat=wine # 都市 × カテゴリ
/jobs?region=bordeaux              # コミュニティ系も同じクエリ規約
/writers?region=bordeaux           # 在住ライター一覧
/explore                           # 「地域から探す」入口 (= region 一覧)
```

- `region` という単一クエリ名を**全リスト系ページで統一**する (city / area / location 等の表記揺れを排除)。
- 「都市選択ピッカー」はグローバルナビには置かず、`/explore` と各リスト系ページのフィルタとして提供する。Paris-as-default を避けるため、初期表示は「すべての地域」とする。
- SEO 観点では `/region/[slug]/articles` のようなパス化も選択肢だが、フィルタの組合せ爆発を避けるためクエリ方式を推奨。

## 6. 優先度付きロードマップ

### Phase 1 (今すぐ / UAT で必要)

- マルチシティ表記が必要な箇所のリラベル (Footer の「Paris 在住」表現、`/explore` サブタイトル等)
- bordeaux / nice のサンプルデータ投入 (本タスクで対応済み)
- `SpotEditor` の座標範囲緩和 (Paris の bounding box 限定をやめ、フランス全土 or 警告のみ)
- 記事フォームに `article.city_id` セレクタを必須項目として明示

### Phase 2 (1-2 週間以内)

- `/region/[slug]` ホームページの充実 (hero / 特集 / 在住ライター / 記事一覧)
- 記事一覧の `region` フィルタ UI 追加
- `SiteFooter` / `SiteHeader` の都市言及を汎用化
- 関連記事ブロックに「同じ都市の他の記事」を追加

### Phase 3 (本格マルチシティ展開時)

- AI Cron の都市別対応 (`parisEvents` → `cityEvents(cityId)`、API パス変更)
- 都市ごとの hero 画像・特集ページ・SEO メタデータの整備
- タイムゾーン対応 (現状は `Europe/Paris` 固定で OK、将来の伊・西展開時に汎用化)
- 都市セレクタ (グローバルピッカー) の導入検討

## 補遺: 監査タスクとして残すもの

本ドキュメントは「現状把握 + 提案」であり、以下はエンジニア側で別途確認するタスク。

1. `SpotEditor.tsx` の座標バリデーション実装 (実際に Paris 範囲限定か?)
2. `articles` テーブルに `city_id` カラムがあるか、UI 側で露出されているか
3. `/region/[slug]` ページの現状実装と SEO メタタグ
4. `parisEvents.ts` / `ai-paris-events` cron の依存関係 (汎用化コスト)
5. `SiteFooter` / `SiteHeader` の Paris 文言の正確な箇所

これらは Phase 1 着手前の「30 分監査」として済ませることを推奨。
